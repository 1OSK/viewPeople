# План разработки ViewPeople

Связано: [[Индекс]], [[Требования и сценарии]], [[Библиотеки и зависимости]], [[Архитектура бэкенд и фронтенд]], [[Сборка Windows exe]].

Ниже — поэтапный план от каркаса до релиза под Windows. Сроки не фиксированы: оценивайте итерациями по 1–2 недели.

## Текущий статус реализации

| Фаза | Состояние |
|------|-----------|
| 1 | Сделано: выбор **двух** камер в режиме «Камера», диалог видео (`src/shared/ipc.ts`, main/preload, `video-source.ts`, UI в `src/renderer/src/ui/app.ts`); пары `deviceId` в настройках (`cameraDeviceId`, `cameraDeviceId2`). |
| 2 | Сделано: таймер с настраиваемым интервалом (200–10000 мс), скрытый canvas, флаг `busy`, время последнего кадра (`pipeline.ts`). |
| 3 | Сделано: **COCO-SSD** + CPU backend в **Web Worker** (`detector.worker.ts`), счётчик «Людей в кадре». |
| 5 (часть) | Сделано: **electron-store** + IPC — интервал, последняя камера, последний видеофайл и `lastSource` для восстановления при старте. |
| 4 (MVP) | Сделано: переключатель **YOLOv8n ONNX** (`onnxruntime-web` WASM) в worker; в **установщике** модель и WASM кладутся в `extraResources` — **YOLO без интернета** у пользователя. Интернет нужен на машине **сборки** (`npm run setup:assets`). Если локальные файлы не найдены — fallback на CDN/URL. |

---

## Фаза 0 — База (уже заложено)

- [x] Репозиторий: Electron + **electron-vite**, TypeScript, скрипты `dev`, `build`, `build:win`.
- [x] Main / preload / renderer с **context isolation**, без `nodeIntegration` в UI.
- [ ] Проверка на целевой машине: `npm run dev`, затем `npm run build:win`, установка из `release/`.

**Критерий готовности:** приложение открывается, нет ошибок в консоли main/renderer.

---

## Фаза 1 — Источник видео (камера + тестовый файл)

**Цель:** единый путь получения «текущего кадра» для дальнейшего анализа.

| Задача | Детали |
|--------|--------|
| UI выбора камеры | Список устройств (`enumerateDevices`), выбор, `getUserMedia`, превью в `<video>`. |
| Режим «видеофайл» | Диалог выбора файла (через IPC + `dialog` в main или drag-and-drop — по политике безопасности). `<video>` + управление play/pause. |
| Абстракция источника | Интерфейс вида: «подписка на кадр / снимок раз в N мс» поверх stream и файла (см. [[Архитектура бэкенд и фронтенд]]). |
| Обработка ошибок | Нет камеры, отказ в разрешениях, неподдерживаемый кодек в файле — понятные сообщения в UI. |

**Критерий готовности:** стабильное превью с камеры и с файла; можно вручную убедиться, что кадр рисуется на Canvas.

---

## Фаза 2 — Пайплайн «1 раз в секунду»

**Цель:** без лишней нагрузки снимать кадр и готовить вход для модели.

| Задача | Детали |
|--------|--------|
| Таймер 1 Гц | `setInterval(1000)` или один таймер + защита от наложения, если инференс дольше 1 с. |
| Canvas / ImageData | `drawImage` с `<video>`, при необходимости масштаб под вход модели. |
| Очередь / флаг «идёт анализ» | Пропуск тика или отмена предыдущего — чтобы не копить задачи. |
| Настройка интервала (опционально) | Поле в UI: 500–2000 мс для отладки. |

**Критерий готовности:** в логе или на экране видно метку времени последнего кадра; CPU не забивается постоянным инференсом.

---

## Фаза 3 — Детекция людей (MVP модель)

**Цель:** число людей в кадре по классу `person`.

| Задача | Детали |
|--------|--------|
| Выбор стека | Старт: **TensorFlow.js + coco-ssd** (см. [[Библиотеки и зависимости]]). |
| Загрузка модели | Один раз при старте; индикатор «модель грузится». |
| Постобработка | Фильтр по классу person, порог confidence; подсчёт боксов после NMS (если даёт модель). |
| Отображение | Крупно: «Людей: N»; опционально overlay боксов на превью. |
| Worker | Вынести инференс в **Web Worker** — сделано: `detector.worker.ts`, передача кадра через **ImageBitmap** + `OffscreenCanvas`. |

**Критерий готовности:** на записи с людьми число в целом правдоподобно; на пустой сцене — 0.

> **Примечание:** первый запуск тянет веса модели из сети (кэш браузера/Electron). Для офлайн-сборки позже — локальные веса / `extraResources`.

---

## Фаза 4 — Качество под «уличную камеру»

**Цель:** лучше работать на мелких фигурах и дистанции.

| Задача | Детали |
|--------|--------|
| Смена модели | **YOLO nano в ONNX** + `onnxruntime-web` или `onnxruntime-node` (см. [[Библиотеки и зависимости]]). |
| Препроцессинг | Letterbox / resize, нормализация как в карточке модели. |
| Тюнинг | Пороги confidence / IoU, сглаживание по времени (опционально) против дрожания счётчика. |
| Регрессионный набор | Набор коротких роликов или кадров для сравнения «до/после». |

**Критерий готовности:** заметное улучшение на ваших тестовых «уличных» кадрах при приемлемом FPS на целевом ПК.

---

## Фаза 5 — IPC, настройки, полировка

**Цель:** удобство использования и расширяемость.

| Задача | Детали |
|--------|--------|
| Расширение preload API | Только необходимые методы: выбор файла, опционально путь к логу. |
| Настройки | Сохранение последнего устройства, пути к файлу, интервала (`electron-store` или простой JSON в userData). |
| Ограничения RTSP / IP | Зафиксировать в UI/доках; при необходимости отдельная задача «мост ffmpeg» (см. [[Требования и сценарии]]). |

**Критерий готовности:** после перезапуска восстанавливаются основные выборы пользователя.

---

## Фаза 6 — Релиз Windows

**Цель:** установщик, пригодный для передачи тестировщикам.

| Задача | Детали |
|--------|--------|
| Иконка, имя продукта | `build.icon`, метаданные (см. [[Сборка Windows exe]]). |
| Тест чистой установки | VM без dev-инструментов; камера и тестовое видео. |
| Подпись (по необходимости) | Authenticode для снижения предупреждений SmartScreen. |
| Модели в дистрибутиве | `extraResources` или загрузка при первом запуске — влияет на размер установщика. |

**Критерий готовности:** `npm run build:win`, установка, запуск, выполнение основного сценария без инструкций «поставьте Node».

---

## Порядок приоритетов (если сжаты сроки)

1. Фаза 1 + Фаза 2 (источник + 1 Гц) — без этого нет продукта.
2. Фаза 3 (coco-ssd) — быстрый end-to-end.
3. Фаза 6 минимально — чтобы рано ловить проблемы сборки.
4. Фаза 4 — когда MVP модели «не тянет» сценарий.
5. Фаза 5 — по мере боли пользователя.

## Риски и точки решений

| Риск | Когда решать |
|------|----------------|
| IP/RTSP камеры «из коробки» нет | После Фазы 1 — отдельный спайк или внешний сервис. |
| Счёт лиц вместо людей | Не смешивать: только класс **person** (см. [[Требования и сценарии]]). |
| Нативные модули (ONNX Node) | Усложняет [[Сборка Windows exe]]; по возможности начинать с web runtime. |
