# Библиотеки и зависимости

Связано: [[Индекс]], [[Архитектура бэкенд и фронтенд]], [[Требования и сценарии]].

## Слой приложения (оболочка)

| Компонент | Назначение | Пакеты / заметки |
|-----------|------------|------------------|
| Desktop shell | Окна, меню, автозапуск, трей | **electron** |
| Сборка `.exe` | Инсталлятор, обновления | **electron-builder** (или electron-forge) |
| Язык | Типобезопасность, экосистема | **TypeScript** |
| Сборка UI | Dev server, бандл рендерера | **vite** + `vite-plugin-electron` / **electron-vite** |

## Захват видео и кадра

| Подход | Где выполняется | Плюсы | Минусы |
|--------|-----------------|-------|--------|
| **Web API** `navigator.mediaDevices.enumerateDevices` + `getUserMedia` | Renderer (Chromium) | Нативно для Electron, без ffmpeg | RTSP «из коробки» нет |
| Элемент **`<video>`** + локальный файл | Renderer | Простой тестовый режим | Кодеки зависят от Chromium |
| **Canvas** `drawImage(video, …)` | Renderer | Единый путь: камера и файл → **ImageData** / тензор | Нужно следить за `readyState` |
| **OpenCV** / **ffmpeg** в main | Node native | Гибко для RTSP, трансcode | Сложнее сборка, больше вес |

Для MVP достаточно **Renderer + getUserMedia + video file**.

## Детекция людей (ключевой выбор)

Задача формулируется как **object detection**, класс **person** (COCO id `1` в многих моделях).

### Вариант A — TensorFlow.js + готовая модель (быстрый старт)

| Пакет | Роль |
|-------|------|
| `@tensorflow/tfjs` | Рантайм |
| `@tensorflow-models/coco-ssd` | Детекция COCO, есть класс `person` |

- **Плюсы:** просто подключить в рендерере или в **Web Worker**, без нативных модулей.
- **Минусы:** на CPU может быть медленнее; для **дальних людей** на уличной камере качество может быть слабее, чем у современного YOLO.

### Вариант B — ONNX Runtime + YOLO (рекомендуется для «камера/улица»)

| Пакет | Роль |
|-------|------|
| `onnxruntime-web` | Инференс в браузере / worker (WASM/WebGL) |
| или `onnxruntime-node` | Инференс в main / worker Node |

- Модель: лёгкая **YOLOv8n** / **YOLOv5n** в ONNX, постобработка NMS своя или из примеров.
- **Плюсы:** хороший баланс скорость/качество на CPU; можно вынести в worker.
- **Минусы:** больше кода препроцессинга (resize, letterbox), контроль версий ONNX opset.

### Вариант C — MediaPipe (Google)

| Пакет | Роль |
|-------|------|
| `@mediapipe/tasks-vision` | Object Detector / др. |

- **Плюсы:** оптимизация под веб, удобные задачи.
- **Минусы:** нужно проверить актуальный список классов и лицензию; не всегда совпадает с привычным COCO «person».

### Вариант D — OpenCV (Node или WASM)

| Пакет | Роль |
|-------|------|
| `opencv4nodejs` / сборки **opencv.js** | Классические детекторы + DNN |

- **Плюсы:** мощно для видео-пайплайнов.
- **Минусы:** тяжёлая сборка под Windows; для людей чаще всё равно нужна **DNN-модель** (YOLO и т.д.).

## Вспомогательные зависимости UI

- **React** / **Vue** / **Svelte** — по предпочтению команды (не обязательны; можно чистый DOM).
- Стили: **CSS modules** / Tailwind — на усмотрение.

## Итоговая рекомендация по этапам

1. **MVP:** Electron + Canvas + **coco-ssd** (TF.js), интервал 1 с, камера + файл.
2. **Улучшение под улицу:** заменить модель на **YOLO nano + onnxruntime-web** (или `onnxruntime-node` в worker thread), подобрать порог confidence / NMS.
