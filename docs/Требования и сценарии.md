# Требования и сценарии

Связано: [[Индекс]], [[Архитектура бэкенд и фронтенд]], [[Библиотеки и зависимости]].

## Функциональные требования

1. **Источник изображения**
   - Выбор **устройства захвата** (веб-камера / IP-камера, если доступна как DirectShow/Media Foundation — см. ограничения ниже).
   - В режиме **камера:** одновременный захват **двух** независимых потоков (два выбранных `deviceId` → два `<video>`, две `MediaStream`). Счёт людей выводится **для каждой камеры отдельно**; общий таймер анализа (интервал) один на обе.
   - **Тестовый режим** и **трансляция по URL** остаются **одноканальными** (один кадр — один источник).
   - **Тестовый режим (файл):** загрузка **видеофайла** вместо живой камеры; кадры берутся из воспроизведения.

2. **Анализ**
   - Снимок и анализ **не чаще 1 раза в секунду** (настраиваемый интервал оставить запасом) — **на канал** в смысле «на камеру»: при двух камерах за один тик **параллельно** обрабатываются кадры с обеих камер (два Web Worker, две копии модели; таймер один на пару).
   - Выход: **целое число — количество людей** в кадре (не идентификация личности). В режиме двух камер — **два числа** (по одному на устройство).

3. **Платформа**
   - Целевой артефакт: **`.exe` для Windows** (x64; при необходимости — отдельная сборка ARM64).

## Нефункциональные требования

- Устойчивость к смене устройства, паузе видео, ошибкам камеры.
- Минимизация нагрузки: инференс **не на каждом кадре видео**, а по таймеру 1 Гц.
- Для уличной камеры: ожидаемы **мелкие фигуры** — нужна модель, обученная на **дальних людей** (см. [[Библиотеки и зависимости]]).

## Ограничения и риски

| Тема | Заметка |
|------|---------|
| Две «физические» камеры | Нужны **два** разных устройства в `enumerateDevices` (или одно + виртуальная, если ОС дублирует). Два одинаковых `deviceId` могут не дать стабильно **два** независимых превью — в UI указывается раздельный выбор «Камера 1 / Камера 2». |
| IP / RTSP камеры | В чистом Chromium/Electron **нет встроенного RTSP**. Варианты: отдельный сервис (ffmpeg → локальный поток), нативный плагин, или камера как **виртуальная DirectShow** на ОС. |
| «Человек» vs «лицо» | Счёт **лиц** ≠ счёт **людей** (спины, каски, дистанция). Для «людей в кадре» лучше **детекция класса person**. |
| Точность | Освещение, окклюзии, толпа — типичные ошибки (слияние боксов, пропуски). В документации к приложению стоит указать **дисклеймер**. |

## Пользовательский сценарий (MVP)

1. Запуск приложения → список устройств.
2. В режиме **Камера:** выбор **двух** устройств (если в системе больше одного) **или** один источник + пустой второй (второе превью без сигнала). **Видеофайл** / **трансляция** — по-прежнему один выбор/URL.
3. Экран превью: при двух камерах — **два** превью; крупно **«Камера 1: N»** и **«Камера 2: M»** (для одного потока — **«Людей: N»**), плюс время последнего анализа.
4. Опционально: лог в файл / экспорт (вне скоупа первой версии — по желанию).
